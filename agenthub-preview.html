Refactor auth module · claude-opus-4 · 14 messages
Can you walk through the session storage path and flag anything that would break under concurrent writes?
CR
Code Reviewer · 2m ago
Sure. Three things stood out. First, the token rotation uses a read-modify-write without a compare-and-swap, so a second request during refresh can overwrite the new token: 
// src/lib/auth/session.ts
async function rotate(userId: string) {
  const current = await store.get(userId);
  const next = sign({ ...current, iat: Date.now() });
  await store.set(userId, next); // race here
}
Second, JWTs are stored in localStorage — switching to an HTTP-only cookie is cheap and removes an XSS exfil path. Third, the refresh window is fixed at 7d, which means any compromise is long-lived; I'd drop to 30m rolling with a server-side revocation list.
Love the first two. On #3 — rolling refresh won't play nicely with our mobile offline mode. What's the minimum we can do and still unblock the compliance ask?
CR
Code Reviewer · just now
Fair. The compliance ask is really about revocability, not TTL. You can keep the 7d token if you add an opaque jti and a revocation list keyed by user — O(1) lookup per request, and you only write on signout / forced rotation. That gets you: same offline behaviour, immediate kill-switch on device loss, and a clean audit trail.

A few concerns:

  1. Default literal vs. expression. If the default is a constant Postgres can now add it in metadata (PG11+). If it's now() or any volatile function, you'll rewrite the whole table and hold an ACCESS EXCLUSIVE lock.
  2. Order matters. Add the column as nullable with default, backfill in batches, then add the NOT NULL constraint with NOT VALID, then VALIDATE CONSTRAINT.
  3. Concurrent writes. New rows inserted during backfill need the app code deployed first so they carry the correct value — otherwise your backfill races the writers.
  4. Rollback. Keep the constraint validation as a separate migration so you can ship the schema change without blocking on it.
-- step 1: nullable with default (metadata-only on PG11+)
ALTER TABLE events
  ADD COLUMN tenant_id uuid DEFAULT '00000000-...';

-- step 2: batched backfill (separate migration)
UPDATE events SET tenant_id = e.owner_tenant
  FROM event_owners e WHERE events.id = e.event_id
  AND events.id BETWEEN $1 AND $2;

-- step 3: validate NOT NULL w/o long lock
ALTER TABLE events ADD CONSTRAINT events_tenant_not_null
  CHECK (tenant_id IS NOT NULL) NOT VALID;
ALTER TABLE events VALIDATE CONSTRAINT events_tenant_not_null;
